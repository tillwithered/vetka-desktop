# Vetka Control MVP — дизайн системы

## Цель

Vetka Control — закрытая внутренняя база товаров Vetka Dolls. Первая версия помогает Виолетте быстро найти куклу Monster High, увидеть её характеристики, источники, актуальные ручные цены и лоты, рассчитать себестоимость и назвать клиенту ориентировочную цену продажи.

Заказы, CRM, Telegram, парсинг маркетплейсов, автоматизация Amazon, сложные роли и AI-функции в V1 не входят.

## Архитектура

- Next.js App Router и TypeScript.
- shadcn/ui preset `b1FjPsUBU` и Tailwind CSS; интерфейс собирается из компонентов shadcn/ui.
- Supabase Auth с закрытым входом по email и паролю. Публичной регистрации нет; пользователей создаёт владелец проекта.
- Supabase Postgres как единственный источник данных.
- Supabase Storage bucket `product-images` для главных фотографий и галерей.
- Server Components для чтения, Server Actions для мутаций, Supabase SSR clients для сессий.
- Row Level Security: авторизованные пользователи могут читать и изменять данные MVP; анонимный доступ запрещён.

## Навигация и экраны

### Вход

Форма email/пароль, состояния загрузки и ошибки. Успешный вход ведёт на Dashboard.

### Dashboard

Показывает количество товаров, товары в поиске, товары с найденной хорошей ценой, последние добавленные товары и последние обновлённые цены. Главные действия: добавить куклу, открыть каталог, лоты или калькулятор.

### Каталог

Карточки с фото, названием, персонажем, линейкой, состоянием, последней закупочной ценой, рекомендованной ценой, прибылью и статусом. Поддерживаются текстовый поиск и фильтры по персонажу, линейке, состоянию, редкости, наличию цены и наличию лотов.

### Создание и редактирование товара

Минимально обязательны название, персонаж, линейка, главное фото и примерный вес. Остальные характеристики необязательны. После создания открывается карточка товара. Редактирование использует ту же форму.

### Карточка товара

Верхняя часть показывает фото, название, статус и основные характеристики. Ниже расположены независимые блоки: галерея, габариты и вес, источники, история ручных цен, лоты и расчёт. Каждый блок имеет локальное действие добавления или изменения.

### Лоты

Общий список ручных лотов с товаром, маркетплейсом, ценой, доставкой, состоянием, статусом и датой проверки. Доступны фильтры по маркетплейсу и статусу.

### Калькулятор

Быстрый самостоятельный расчёт и тот же расчёт внутри карточки товара. Вводятся цена товара, доставка по США, курс USD/KZT, вес, тариф доставки за килограмм, дополнительные расходы и цена продажи.

## Модель данных

Все таблицы содержат UUID `id`, `created_at` и при необходимости `updated_at`.

### products

Каноническая карточка товара: `name`, `character_name`, `line_name`, `brand`, `release_year`, `category`, `condition_type`, `rarity_level`, `status`, `description`, `main_image_path`, `box_width_cm`, `box_height_cm`, `box_depth_cm`, `loose_weight_kg`, `boxed_weight_kg`, `default_weight_kg`, `notes`, `search_keywords`.

### product_images

Галерея: `product_id`, `storage_path`, `alt_text`, `position`.

### product_sources

Ручные источники: `product_id`, `source_type`, `url`, `comment`.

### product_prices

История ручных проверок цен: `product_id`, `source_type`, `condition_type`, `price`, `shipping_price`, `currency`, `price_kzt`, `comment`, `checked_at`. Структура пригодна для будущего автоматического заполнения.

### lots

Конкретные предложения: `product_id`, `marketplace`, `url`, `title`, `image_url`, `price_usd`, `shipping_usd`, `condition_type`, `seller_name`, `status`, `comment`, `checked_at`.

### shipping_profiles

Варианты доставки: `product_id`, `service_name`, `weight_kg`, `price_per_kg_usd`, `shipping_cost_usd`, `shipping_cost_kzt`, `comment`.

### product_calculations

Сохранённые расчёты: `product_id`, `buy_price_usd`, `local_shipping_usd`, `international_shipping_usd`, `usd_to_kzt_rate`, `extra_costs_kzt`, `total_cost_kzt`, `recommended_sale_price_kzt`, `profit_kzt`, `profit_percent`.

## Формулы

`international_shipping_usd = weight_kg × price_per_kg_usd`, если значение не введено вручную.

`total_cost_kzt = (buy_price_usd + local_shipping_usd + international_shipping_usd) × usd_to_kzt_rate + extra_costs_kzt`.

`profit_kzt = recommended_sale_price_kzt - total_cost_kzt`.

`profit_percent = profit_kzt / total_cost_kzt × 100`; при нулевой себестоимости процент не рассчитывается.

Все денежные вычисления на сервере повторяют клиентский preview. В базе денежные значения хранятся как `numeric`, а не float.

## Поведение и ошибки

- Формы валидируются на клиенте и сервере.
- Ошибки Supabase показываются рядом с действием через Alert/Field и не очищают введённые значения.
- Фото проверяются по типу и размеру; неудачная загрузка не создаёт битую запись товара.
- Удаление товара требует подтверждения и каскадно удаляет связанные записи; файлы Storage удаляются отдельным безопасным действием.
- Пустые состояния всегда предлагают следующее действие.
- Внешние ссылки открываются в новой вкладке с безопасными атрибутами.

## Визуальный подход

Интерфейс следует shadcn preset без отдельной самописной библиотеки компонентов. Компоновка — спокойная рабочая база: sidebar, PageHeader, Cards, Tabs, Forms, Dialog/Sheet, Badge, DropdownMenu и Skeleton. Фотография куклы остаётся главным визуальным якорем. Русские подписи короткие, значения денег форматируются в USD и KZT.

## Проверка

- Unit-тесты формул и схем валидации.
- Интеграционные тесты серверных операций с подменённым Supabase client.
- Smoke-проверка маршрутов и защищённого layout.
- Ручная проверка ключевого сценария: вход → создание товара с фото → добавление источника, цены и лота → расчёт → поиск товара в каталоге.
- `pnpm lint`, `pnpm test` и `pnpm build` должны завершаться успешно.

## Критерий готовности

V1 готова, когда авторизованный пользователь может создать и найти куклу, загрузить и увидеть её фото, заполнить характеристики и размеры, вручную добавить источники, цены и лоты, получить и сохранить расчёт себестоимости и прибыли, а Dashboard отражает эти данные.
